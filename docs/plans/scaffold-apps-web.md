# Scaffold `apps/web` — the main Angular app

## Context

Craftsman's Ledger's `apps/` and `libs/` directories are currently empty; [ADR-0001](../adr/0001-moonrepo-mise-pnpm-for-monorepo-tooling.md) planned for two Angular clients and a NestJS API but deferred actually wiring any of them up. This plan scaffolds the first one — the main web client — as a fully self-contained, moon-orchestrated Angular CLI workspace, and builds out the two shared tooling packages (`eslint-config`, `stylelint-config`) that were left as empty placeholders waiting for exactly this moment.

Every decision below was resolved in a `/grill-with-docs` session and is recorded as [ADR-0012](../adr/0012-self-contained-angular-workspace-per-app.md) through [ADR-0019](../adr/0019-manual-tailwind-at-rule-allowlist.md). This plan is purely about sequencing the implementation — no further design decisions are expected, only CLI-version-specific flag names to confirm at execution time.

**Explicitly out of scope:** domain routes/components (empty shell app only), data-loading/API integration (`data/medieval_dynasty.xlsx` stays unwired), and E2E tests ([ADR-0013](../adr/0013-zoneless-vitest-testing-stack.md) names Playwright as the agreed tool, but building the suite is deferred).

**Delivery:** each phase below ships as its own pull request against `main`, rather than one large PR for the whole app.

## Phase 1 — Scaffold `apps/web` and register it with moon

- Run the Angular CLI (`pnpm dlx @angular/cli@latest new`) targeting `apps/web`: routing enabled, SCSS stylesheets, no SSR/prerendering, default `app` component prefix, unscoped package name `web`. Confirm at execution time whether the installed CLI version can select Vitest as the unit-test runner directly during `ng new` (skips redoing test setup in Phase 6) — use it if offered, otherwise accept the default and replace it in Phase 6.
- Enable zoneless change detection in `apps/web/src/app/app.config.ts` (the stable `provideZonelessChangeDetection` API, or whatever the installed Angular version currently calls it — check its docs, since this API name has moved between "experimental" and stable naming across recent majors).
- Move every `@angular/*` dependency plus `typescript` out of the generated `apps/web/package.json` and into a new `catalog:angular` entry in `pnpm-workspace.yaml` (mirroring the existing `catalog:tooling` shape), using the exact versions `ng new` resolved. Point `apps/web/package.json` at `catalog:angular` for those entries, per [ADR-0012](../adr/0012-self-contained-angular-workspace-per-app.md).
- Add `apps/web/moon.yml` (`layer: "application"`, `stack: "frontend"`, `toolchains: ["node"]`, modeled on `packages/tsconfig/moon.yml`) with `start` (`ng serve`) and `build` (`ng build`) tasks. `test`, `test-ci`, `lint-ts`, `lint-css` are added in their own phases below once the underlying tools exist.
- Verify: `pnpm install`, `pnpm moon run web:build`, `pnpm moon run web:start` (check the dev server actually serves the default Angular CLI landing page in a browser).

## Phase 2 — Tailwind v4 + SCSS

- Add Tailwind v4 (`tailwindcss`, `@tailwindcss/postcss`) as direct devDependencies of `apps/web` (not `catalog:angular` — Tailwind isn't an Angular package, no shared-catalog decision was made for it).
- Wire Tailwind into the Angular build (PostCSS config consumed by `@angular/build`) and add `@import "tailwindcss";` to the app's global stylesheet. Per [ADR-0014](../adr/0014-tailwind-utility-first-styling.md), this is the *only* global SCSS Tailwind needs — component templates use Tailwind utilities directly, SCSS stays reserved for real global concerns.
- Add `prettier-plugin-tailwindcss` to the root `package.json` devDependencies and `.prettierrc.yaml`'s `plugins` list, so class sorting happens through the existing repo-wide Prettier run ([ADR-0005](../adr/0005-root-only-prettier-config.md)/[ADR-0006](../adr/0006-prettier-owns-formatting-boundary.md)).
- Verify: `pnpm moon run root:format` sorts Tailwind classes in a scratch template edit; `pnpm moon run web:build` still succeeds with Tailwind's generated CSS present in the output.

## Phase 3 — `packages/tsconfig` Angular overlay

- Add `packages/tsconfig/angular.json`, extending `base.json` with the `target`/`module`/`moduleResolution`/`lib` settings Angular's installed TypeScript version needs (per [ADR-0011](../adr/0011-per-framework-typescript-catalogs.md), `base.json` deliberately omits these).
- Point `apps/web/tsconfig.json` (and its app/spec variants) at `@craftsmans-ledger/tsconfig/angular.json` instead of Angular CLI's generated inline `compilerOptions`; add `@craftsmans-ledger/tsconfig` as a `workspace:*` dependency of `apps/web`.
- Verify: `pnpm moon run web:build` still succeeds with the swapped-in tsconfig.

## Phase 4 — `packages/eslint-config` (base + angular overlay), a root-level config, and `apps/web` linting

- All ESLint config files added in this phase use the `.mts` extension uniformly (root `eslint.config.mts`, `packages/eslint-config/base.mts` and `angular.mts`, `apps/web/eslint.config.mts`), authored with ESLint's `defineConfig`/`globalIgnores` helpers. `.mts` is unambiguous ESM regardless of a package's own `package.json` `"type"` field, notably sidestepping adding `"type": "module"` to the Angular-CLI-generated `apps/web/package.json`. Node 24.18.0 (this repo's pinned version) natively strip-types `.mts` with no flag needed, but `jiti` (`catalog:tooling`) is installed as a devDependency wherever the `eslint` CLI runs (root, `apps/web`) regardless.
- New packages needed for this phase: `eslint`, `@eslint/js`, `eslint-config-prettier`, `typescript-eslint`, `jiti`, and a `catalog:tooling`-local `typescript` (separate from `catalog:angular`'s pin — it only ever backs `packages/tsconfig/node.json`-based type-checking of plain Node config files, never a framework's actual app source, so it can't collide with a future `catalog:nest`-style fork) go in `catalog:tooling`; `angular-eslint` goes in `catalog:angular` ([ADR-0021](../adr/0021-typescript-eslint-stays-in-tooling-catalog.md) explains why `typescript-eslint` doesn't fork per-framework like raw `typescript` does).
- Add `packages/tsconfig/node.json`, a new overlay (`target`/`module`/`moduleResolution: "nodenext"`, `noEmit`, `allowImportingTsExtensions` — needed since `.mts` files import each other with explicit extensions under `nodenext`) for plain Node ESM packages authoring their own TypeScript, alongside the existing `angular.json` overlay.
- Flesh out `packages/eslint-config` from its current `.gitkeep` placeholder: `package.json` + `moon.yml` (`layer: "configuration"`, mirroring `packages/tsconfig`), matching README.md. `@eslint/js`, `typescript-eslint`, `eslint-config-prettier`, and `angular-eslint` are plain `dependencies` of this package (not `peerDependencies` — that convention exists for externally-published shareable configs managing duplicate installs across consumers; this package is `"private": true` and never leaves the pnpm workspace, where the catalogs already guarantee one version repo-wide).
- `base.mts` (ESM flat config): `@eslint/js` recommended, `typescript-eslint`'s `recommendedTypeChecked` + `stylisticTypeChecked`, and `eslint-config-prettier` — exported via the package's `./base` subpath export ([ADR-0017](../adr/0017-typescript-eslint-recommended-type-checked.md)). Also holds, because they're framework-agnostic conventions rather than Angular-specific ones ([ADR-0016](../adr/0016-base-overlay-split-for-lint-configs.md)'s reasoning applied one level deeper):
    - `globalIgnores` for `dist/**`, `coverage/**`, `reports/**` (generic build/test-output directories any future app will also produce).
    - An override scoped to `**/*.spec.ts` relaxing `@typescript-eslint/no-explicit-any` and `@typescript-eslint/no-non-null-assertion` (a generic test-code convention, not Angular-specific).
    - An override scoped to `**/*.{js,mjs,cjs}` applying `tseslint.configs.disableTypeChecked` — plain JS files (like this repo's own root-level config files) aren't covered by any tsconfig `include`, so the type-checked rule tiers can't run against them.
- `angular.mts`: imports and spreads `base.mts` internally, adds `angular-eslint`'s recommended rules, its template-recommended rules, and its template accessibility rules — exported via `./angular` (exact config export names to confirm against whatever `angular-eslint` version installs at implementation time). Also holds a `globalIgnores` for `.angular/**` (Angular CLI's own build cache, no equivalent in `base.mts`).
- `packages/eslint-config` needs its own `tsconfig.json` (extending `@craftsmans-ledger/tsconfig/node.json`, `include: ["*.mts"]`, explicit `"types": ["node"]` — automatic `@types` acquisition doesn't reliably resolve `@types/node`'s ambient `ImportMeta` augmentation in this pnpm monorepo) and its own `eslint.config.mts` (self-lints the package's own `.mts` files, wiring `projectService`/`tsconfigRootDir` via `dirname(fileURLToPath(import.meta.url))` — not `import.meta.dirname` directly, which needs the same `@types/node` augmentation and offers no advantage here).
- `apps/web/eslint.config.mts` imports `@craftsmans-ledger/eslint-config/angular` and re-exports it, adding `parserOptions: { projectService: true, tsconfigRootDir: dirname(fileURLToPath(import.meta.url)) }`, scoped to `files: ['**/*.ts']` (an unscoped block would clobber `base.mts`'s `disableTypeChecked` exemption for the `.mts` config file itself) — the shared package deliberately leaves `project`/`projectService` unset, mirroring how `packages/tsconfig/base.json` omits environment-specific compiler options. Since `apps/web/tsconfig.json` is a solution-style config (`"files": []`, `references` only) and `eslint.config.mts` isn't covered by `tsconfig.app.json`/`tsconfig.spec.json`, add a sibling `apps/web/tsconfig.eslint.json` (same `node.json`-overlay shape as `packages/eslint-config`'s) and reference it from `apps/web/tsconfig.json`'s `references` array — this keeps the solution config in "files: [] + references" form, which is exempt from TypeScript's `--build`-mode requirement that referenced projects be `composite: true`. Add `@craftsmans-ledger/eslint-config` as a `workspace:*` devDependency of `apps/web`, alongside the existing `@craftsmans-ledger/tsconfig` entry.
- Add the `lint-ts` task to `apps/web/moon.yml` (`eslint .`; `inputs`: `src/**/*`, `eslint.config.mts`, `tsconfig*.json` — mirroring the existing `build` task's shape, no explicit `outputs`).
- New: add a root-level `eslint.config.mts`, importing `@craftsmans-ledger/eslint-config/base` only (no Angular concerns at the repo root). Scoped with single-star `files` globs (`*.js`, `*.mjs`, `*.cjs`, `*.ts`) so it only ever matches root-level files (`commitlint.config.mjs`, `lint-staged.config.mjs` today), plus an explicit `ignores: ['apps/**', 'libs/**', 'packages/**']` as a safety precaution even though the single-star globs already can't cross directory boundaries. Add a matching `lint-ts` task to the root `moon.yml`. Also add a root `tsconfig.json` (same `node.json`-overlay shape, `include: ["*.mts"]`) so the root `eslint.config.mts` gets real `tsc` coverage too.
- Add a `typecheck` task alongside `lint-ts` in `packages/eslint-config`, `apps/web`, and the root `moon.yml`: ESLint's own type-aware rules don't catch every real `tsc` error (an `exactOptionalPropertyTypes` conflict and an `ImportMeta` typing gap both slipped past `lint-ts` and only surfaced as IDE diagnostics). `packages/eslint-config` and root run `tsc --noEmit -p tsconfig.json` directly; `apps/web` needs a `script:` (not `command:` — moon's `command`/`args` don't support shell chaining) running `tsc --noEmit -p tsconfig.app.json && tsc --noEmit -p tsconfig.spec.json && tsc --noEmit -p tsconfig.eslint.json` sequentially, since its solution-style `tsconfig.json` can't be type-checked directly via `tsc -b` (a pre-existing, unrelated `tsconfig.spec.json` `rootDir`/`TS5011` issue blocks build-mode) nor as a single `-p` (it has no `files`/`include` of its own).
- Verify: `pnpm moon run web:lint-ts`, `root:lint-ts`, `eslint-config:typecheck`, `web:typecheck`, and `root:typecheck` all run clean; deliberately introduce a lint violation and a type error in each scope to confirm the rule sets and `tsc` actually fire, then revert them.

## Phase 5 — `packages/stylelint-config` (base + angular overlay) and `apps/web` style linting

- Flesh out `packages/stylelint-config` the same way: `package.json` + `moon.yml` + README.md.
- `base.js`: `stylelint-config-standard-scss`, `stylelint-config-clean-order` ([ADR-0018](../adr/0018-stylelint-clean-order-over-recess-order.md)), plus disabling of any Stylelint rules that conflict with Prettier's SCSS formatting — exported via `./base`.
- `angular.js`: extends `base.js` internally, adds a manual `at-rule-no-unknown` override allowlisting Tailwind v4's at-rules (`@theme`, `@utility`, `@variant`, `@apply`, `@plugin`, `@source`) per [ADR-0019](../adr/0019-manual-tailwind-at-rule-allowlist.md) — exported via `./angular`.
- `apps/web`'s Stylelint config extends `@craftsmans-ledger/stylelint-config/angular`.
- Add the `lint-css` task to `apps/web/moon.yml` (`stylelint "**/*.scss"`).
- Verify: `pnpm moon run web:lint-css` runs clean against the app's real Tailwind `@import`/`@theme` usage from Phase 2 (this is the real test of the at-rule allowlist), and against any hand-written SCSS.

## Phase 6 — Vitest testing stack

Phase 1 already scaffolded Vitest (`vitest`, `jsdom` devDependencies; `tsconfig.spec.json`'s `vitest/globals`; a bare `test: { builder: "@angular/build:unit-test" }` target in `angular.json`) — there's nothing to remove. This phase adds the remaining devDependencies, wires `angular.json`'s `test` target, and adds a narrowly-scoped `vitest.config.ts`. The config split below (what lives in `angular.json` vs. the standalone file) is settled by [ADR-0022](../adr/0022-vitest-config-split-angular-json-vs-vitest-config-ts.md) — read it before touching either file, since the split isn't a style choice, it avoids real bugs (silent duplicate test runs, settings that get overridden with no visible warning).

- Add `@vitest/browser-playwright` (not `@vitest/browser` — Vitest 4 splits browser-mode support into per-provider packages), `@vitest/coverage-v8`, and `@vitest/ui` as devDependencies of the **root** `package.json` (not `apps/web/package.json`) — `@angular/build`'s unit-test builder dynamically `import()`s these relative to the workspace root, per [ADR-0020](../adr/0020-root-level-angular-workspace.md)'s extended consequence. `playwright` itself is a devDependency of `apps/web/package.json`, since it's only ever invoked directly from that project.
- `apps/web/vitest.config.mts` (`.mts`, matching `eslint.config.mts`'s convention, not `.ts`) — scoped to only what `angular.json`'s builder schema can't express: `test.browser.screenshotFailures: false` (disable default failure screenshots), `test.coverage.reportsDirectory: 'coverage'`, and an explicit `test.root: import.meta.dirname` so that path always resolves to `apps/web/coverage` regardless of the invoking process's cwd. Do **not** set `test.browser.instances`/`name`, `test.reporters`, `test.coverage.reporter`, or `test.watch` here — see [ADR-0022](../adr/0022-vitest-config-split-angular-json-vs-vitest-config-ts.md). Being `.mts`, it needs the same follow-on wiring `eslint.config.mts` already has: add it to `apps/web/tsconfig.eslint.json`'s `include` (typecheck coverage) and to `packages/eslint-config/base.mts`'s `disableTypeChecked` file list alongside `eslint.config.{ts,mts,cts}` (ESLint's type-aware rules are skipped for both); `apps/web/tsconfig.spec.json` also needs `"files": ["vitest.config.mts"]` added, since the builder's own test-execution tsconfig (not just the ESLint-focused one) must cover it too.
- `angular.json`'s `web` project `test` target:
    - Base `options`: `browsers: ["chromium"]` (per [ADR-0013](../adr/0013-zoneless-vitest-testing-stack.md)); `headless: true` (always headless — local dev gets the interactive experience from the Vitest UI dashboard instead of a visible browser window); `runnerConfig: "apps/web/vitest.config.mts"` (explicit path, not `runnerConfig: true` — auto-discovery only looks for `vitest-base.config.*`); `include: ["**/*.spec.ts"]` (this repo only uses the `.spec.ts` naming convention, narrower than the schema's default which also matches `**/*.test.ts`); `tsConfig: "apps/web/tsconfig.spec.json"`; `coverage: true` via the `v8` provider; `coverageReporters: ["html", "text-summary"]`; `coverageExclude: ["src/main.ts", "src/*/config/**"]` (bootstrap code and per-module routing/DI config — better covered by E2E than unit tests); `coverageThresholds` at `80%` for statements/branches/functions/lines, aggregate (no `perFile`).
    - `defaultConfiguration: "development"`, with two `configurations`:
        - `development`: `reporters: ["dot", ["html", { "outputFile": "apps/web/reports/index.html" }]]` (the builder's schema documents this reporter as `dots`, but Vitest 4.1.9's actual `ReportersMap` key is `dot`, singular — trust the runtime, not the stale schema description), plus `ui: true` and `watch: true` — the `outputFile` is set via the reporters tuple form, since plain `test.outputFile` isn't valid to set from `vitest.config.mts`, it's a Vitest root-only option that doesn't survive the builder's per-project config nesting.
        - `ci`: same `reporters` plus `github-actions`, `watch: false`. `angular.json` is static, so this uses a real `configurations` entry rather than a `process.env.CI` runtime check — `test-ci` always targets this configuration explicitly, since that task only ever runs in CI anyway.
- `apps/web/moon.yml`: `test` (`command: 'ng run web:test'`, `options: { runInCI: false }` — interactive watch mode, never runs in CI) and `test-ci` (`command: 'ng run web:test:ci'`, `inputs` mirroring `build`'s shape plus `vitest.config.mts`, `outputs: ['coverage', 'reports']` for caching). Both depend (`deps: ['install-browsers']`) on a new shared task rather than documenting the Playwright browser install as a manual step — see [ADR-0023](../adr/0023-stack-scoped-moon-tasks-for-environment-provisioning.md): `.moon/tasks/frontend.yml` defines `install-browsers` (`playwright install chromium`) inherited by any project with `stack: 'frontend'`. `lint-ts` and `typecheck` also get `vitest.config.mts` added to their `inputs`, matching how they already track `eslint.config.mts`.
- Verify: `pnpm moon run web:test` opens the Vitest UI dashboard in watch mode against headless Chromium; `pnpm moon run web:test:ci` runs once, exits, and produces an `html`+`text-summary` coverage report under `apps/web/coverage` plus an `html` test report at `apps/web/reports/index.html` (including a `github-actions`-annotated failure line); deliberately break a test to confirm `test-ci` actually fails, then revert — confirmed working end to end, including the full `pnpm moon ci :format-check :build :test-ci :lint-ts :lint-css :typecheck` target list.

## Phase 7 — Wire linting/testing into git hooks and CI

- Extend `lint-staged.config.mjs` with `*.ts`/`*.scss` entries running `eslint --fix`/`stylelint --fix` respectively, alongside the existing catch-all `"*": "prettier -w"` rule. [ADR-0004](../adr/0004-husky-lint-staged-for-git-hooks.md) explicitly named this as deferred "until real code exists to lint" — that condition is now met.
- Extend `.github/actions/ci/action.yml`'s `pnpm moon ci :format-check` call to `pnpm moon ci :format-check :build :test-ci :lint-ts :lint-css :typecheck`.
- Add a Playwright Chromium install step to the CI setup (`.github/actions/setup/action.yml` or the `ci` action) so `test-ci`'s browser-mode Vitest run has a browser available.
- Verify: push a throwaway branch (or inspect via `act`/manual reasoning) confirming the composite action's step order provisions Playwright before `moon ci` runs; run `pnpm lint-staged` locally against a staged dummy change to confirm ESLint/Stylelint actually fire.

2. `pnpm moon run web:build`, `web:start`, `web:test`, `web:test-ci`, `web:lint-ts`, `web:lint-css`, `web:typecheck`, `root:lint-ts`, `root:typecheck`, `eslint-config:typecheck` all succeed individually.
3. `pnpm moon run web:start`, then open the app in a browser and confirm the default Angular landing page renders with Tailwind's base styles applied.
4. `pnpm moon run root:format-check` passes, confirming Tailwind class sorting and the rest of the workspace formatting stay consistent.
5. `pnpm moon ci :format-check :build :test-ci :lint-ts :lint-css :typecheck` (the same target list CI will run) passes locally before pushing.
