# Use husky and lint-staged for git hooks, not moon's native vcs.hooks

Moonrepo can manage git hooks natively via `vcs.hooks`/`vcs.sync` in `.moon/workspace.yml`, and combined with `--affected --status=staged` and a task's `options.affectedFiles: true`, it can match lint-staged's core behavior of running a linter against only staged files. We chose husky and lint-staged anyway: lint-staged also re-stages fixed files automatically (`git add` after autofix), which moon has no native equivalent for ā€” a moon-only hook would need to chain a manual `git add` step to get the same result. Moon's `--affected`/`--status` filtering still has a job elsewhere, e.g., CI runs via `moon ci`, which need to check everything affected by a whole PR rather than just the staging area.

Husky and lint-staged were installed once commitlint ([ADR-0010](./0010-commitlint-via-cli-in-ci.md)) needed a local counterpart to its CI-only enforcement, ahead of any real app/lib code existing to lint. `.husky/pre-commit` runs `pnpm lint-staged`, whose config lives in `lint-staged.config.mjs` ā€” a separate file, mirroring `commitlint.config.mjs`'s style rather than an inline `package.json` field. That config is a single `"*": "prettier -w"` rule: it hands every staged file to Prettier and lets `.prettierignore`'s existing deny-by-default/allowlist pattern (json/yaml/yml/js/mjs/cjs/ts) decide what actually gets formatted, rather than duplicating that extension list as a second source of truth inside lint-staged's own config. No `--ignore-unknown` flag is needed: since `.prettierignore` denies everything by default and only re-includes known types, a staged file of an unrecognized type is already excluded before Prettier would need to infer a parser for it. lint-staged calls Prettier directly instead of going through moon's `format`/`format-check` tasks ā€” those stay whole-repo tools for CI and manual runs, uninvolved in the git-hook path, since lint-staged's re-staging behavior already depends on invoking the formatter itself with an explicit, staged-only file list. `.husky/commit-msg` runs `pnpm commitlint -e "$1"` directly, giving local enforcement alongside ADR-0010's CI check. Both hook commands invoke the local binary via bare `pnpm <bin>` ā€” pnpm resolves workspace binaries directly without needing `exec` or `npx`, consistent with this repo's pnpm-only tooling policy (`devEngines.packageManager`).

## Consequences

- `vcs.hooks`/`vcs.sync` in `.moon/workspace.yml` stay unset ā€” husky owns hook installation, not moon.
- Husky and lint-staged are installed and wired now, ahead of real app/lib code existing ā€” driven by commitlint ([ADR-0010](./0010-commitlint-via-cli-in-ci.md)) needing a local hook, not by a lint target being ready. Staged-file tooling beyond Prettier (e.g. ESLint, once `packages/eslint-config` is in use) stays deferred until real code exists to lint.
- Adding a new staged-file check later means adding a key to `lint-staged.config.mjs`, not a new moon task ā€” the git-hook path and the moon-task path are deliberately kept separate.
