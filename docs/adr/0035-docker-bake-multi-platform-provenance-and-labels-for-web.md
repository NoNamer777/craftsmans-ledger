# `web`'s CI image build uses Docker Bake, targets amd64+arm64, and attaches full provenance

`.docker/web/docker-bake.hcl` replaces a plain `docker build`/`docker/build-push-action` invocation as the CI build mechanism for `web`'s image. It defines an empty `docker-metadata-action` target that a `build` target inherits from, following [`docker/metadata-action`'s documented Bake integration](https://github.com/docker/metadata-action#bake-definition): `docker/metadata-action` emits a bake-definition file (tags, OCI labels) that's passed to `docker/bake-action` alongside this repo-owned file via its `files:` input, so tags and labels never have to be hand-duplicated between the two.

The build targets `linux/amd64` and `linux/arm64`, orchestrated by individually pinned actions — `docker/setup-qemu-action`, `docker/setup-buildx-action`, `docker/metadata-action`, `docker/bake-action` — rather than the newer `docker/github-builder` reusable workflow. `github-builder` distributes each platform to a natively matched runner instead of emulating via QEMU, but as of this writing it's thinly documented and comparatively unproven; the individually wired actions are the long-established, widely used pattern, and QEMU's emulation cost is negligible here since `web`'s image is a small static site served by nginx, not a compute-heavy build.

`docker/setup-buildx-action` and the `docker/login-action` GHCR login step are factored into a composite action, `.github/actions/docker/`, shared by `pull-request.yml`'s `docker` job and `push-main.yml`'s `docker-promote` job — the same duplication-avoidance already applied to affected-detection (see [ADR-0036](./0036-moon-affected-gates-docker-jobs.md)). `docker/setup-qemu-action` stays outside it: only `docker`'s actual Bake build needs emulation for the non-native platform; `docker-promote` never builds anything (it retags and deletes existing tags), so bundling QEMU setup there would be pure waste.

The composite action takes an explicit, required `token` input rather than reading `${{ github.token }}` internally. Composite actions can't reference the `secrets` context directly (it's blocked for security reasons, unlike JavaScript/Docker actions) — but `github.token` is a functionally identical value exposed through the `github` context, which *is* accessible directly inside a composite action's own steps, so reading it internally would have worked and avoided callers needing to pass anything. An explicit input was chosen anyway: every caller states in its own `with:` block exactly which token the action authenticates with (`${{ secrets.GITHUB_TOKEN }}`, matching how `dataaxiom/ghcr-cleanup-action` is already invoked elsewhere in these same workflows), rather than that detail being invisible inside the composite action's implementation. It also leaves the door open for a future caller to pass a different token (e.g., a PAT with broader package permissions) without changing the action itself.

The `attest` setting on the `build` target is `type=provenance,mode=max` — full build provenance (build definition, source materials, environment), rather than the slimmer `mode=min`. Since this repo and its images are already public, there's no confidentiality reason to prefer the slimmer record, and full provenance gives any future consumer (e.g., a deploy pipeline, or #94's eventual E2E tooling) complete supply-chain traceability.

`.docker/web/Dockerfile`'s `base` stage now pins `FROM --platform=$BUILDPLATFORM`, so the entire `pnpm install` / `moon docker scaffold+setup` / `ng build` chain always runs natively on the CI runner's own architecture rather than under QEMU emulation — safe here because Angular's build output (`apps/web/dist/browser`) is plain JS/HTML/CSS, architecture-independent, so there's no cross-compilation concern the way there would be for a compiled binary. The `runtime` stage (`FROM nginx:1.29-alpine...`) is deliberately left without a `--platform` override, so it resolves per actual target platform for each of Bake's matrix entries — its only work is copying static files and `chown`, trivial regardless of emulation.

## Consequences

- Local/manual builds (`docker build -f .docker/web/Dockerfile -t craftsmans-ledger-web .`, as documented in the Dockerfile's own header) still work unchanged and single-platform; Bake is CI-only.
- A future app's Dockerfile that does perform architecture-specific compilation (e.g., a Go or Rust binary) cannot copy this Dockerfile's `--platform=$BUILDPLATFORM` placement uncritically — it would need `TARGETARCH`/`TARGETOS` build args and actual cross-compilation logic, not just the pin.
- Re-evaluate `docker/github-builder` once it has a longer track record; nothing here precludes migrating to it later if QEMU's cost stops being negligible (e.g., a future app with a heavier native build step).
