# `next` is a retag of the already-built `pr-<N>` image, not a rebuild from `main`

`pull-request.yml`'s `docker-web` job pushes `web`'s multi-platform image to `ghcr.io/nonamer777/craftsmans-ledger/web` tagged `pr-<N>` (via `docker/metadata-action`'s `type=ref,event=pr` rule, which produces exactly that format). When the PR merges, `push-main.yml`'s `docker-promote` job promotes it to `next` with `docker buildx imagetools create --tag ...:next ...:pr-<N>` — copying the existing manifest list, not rebuilding. This is the standard "build once, promote" pattern: the artifact that ships as `next` is byte-for-byte identical to what was actually built (and, once #94 lands, tested) during the PR, rather than a fresh build from `main` that could differ due to dependency drift or non-determinism despite nominally being "the same code."

The retag itself is wrapped in a composite action, `.github/actions/promote/`, taking `package`, `source-tag`, and `target-tag` inputs — `docker-promote` calls it with `package: ${{ env.WEB_DOCKER_IMAGE }}` (the full `ghcr.io/nonamer777/craftsmans-ledger/web` ref; see [ADR-0044](./0044-hardcode-ghcr-owner-for-web.md) for why that's a hardcoded literal), `source-tag: pr-<N>`, `target-tag: next`. Unlike `.github/actions/cleanup/` ([ADR-0038](./0038-tag-cleanup-via-maintained-ghcr-cleanup-action.md)), it doesn't take a separate `owner` input — its underlying `docker buildx imagetools create` just needs one complete ref per side of the retag, so `package` carries the registry and owner inline rather than being assembled from parts. There's only one call site today, so this is currently more about naming the operation and keeping the workflow YAML declarative than removing duplication — the latter benefit arrives if/when a second app needs the same promote-by-retag step.

The job resolves which PR to promote by calling the GitHub REST API (`GET /repos/{owner}/{repo}/commits/{sha}/pulls`) rather than regex-parsing `github.event.head_commit.message` for a PR number. This repo's merge settings (squash and rebase merges both disabled, only true merge commits allowed) currently make the commit-message shape (`Merge pull request #<N> from ...`) reliable enough for a regex, but that reliability is contingent on a repository setting that could change independently of this workflow; the API call carries the same guarantee without that dependency, at the cost of one extra request per push to `main`.

Immediately after the retag succeeds, the same job removes the now-redundant `pr-<N>` tag (via the mechanism in [ADR-0038](./0038-tag-cleanup-via-maintained-ghcr-cleanup-action.md)) — there's no window where both tags exist pointing at duplicate content, and no separate workflow run is needed to clean it up.

`push-main.yml` has a single **workflow-level** concurrency group (`${{ github.workflow }}-${{ github.ref }}`, `cancel-in-progress: false`) covering both `ci` and `docker-promote` together, unlike `pull-request.yml`'s workflow-level group, which does cancel superseded runs. On a PR, cancelling a superseded run is correct: a newer push to the same PR branch produces the same `pr-<N>` tag, so the older run being canceled just means the newer commit's build is what ends up published — nothing is lost. On `push-main.yml`, each push is a distinct, one-shot promotion opportunity for a specific merged PR that never recurs, so a canceled run has a lasting effect: if two merges land close together and the first run is canceled before its retag step, that PR's image is *permanently* skipped — nothing re-runs for it later, since the next push's moon-affected check only diffs against the commit the first push already landed.

`ci` needed the same non-cancelling treatment as `docker-promote`, not just the latter: `ci` now also computes `web-affected` (see [ADR-0036](./0036-moon-affected-gates-docker-jobs.md)), which `docker-promote` depends on via `needs: ci`. A `needs:` dependency on a canceled job is treated as unmet, so cancelling `ci` doesn't just skip re-validating that push — it silently skips that push's entire promotion and cleanup too, the same permanent-gap failure mode fixing only `docker-promote`'s own concurrency would have missed.

Both jobs' non-cancelling groups were initially kept separate (`ci`'s own group, `docker-promote`'s own group) rather than merged into one. Separate groups still preserve correct cross-push ordering in practice: since `ci` runs are serialized among themselves, an earlier push's `docker-promote` always becomes eligible before a later push's `docker-promote` can even exist, so the later one queues behind it in `docker-promote`'s own group. But that ordering depends on GitHub's FIFO handling of concurrency queues, which [GitHub's own documentation qualifies as intent, not a guarantee](https://docs.github.com/en/actions/using-jobs/using-concurrency) ("actual start time... may vary, ordering is not guaranteed"). The two groups were consolidated into one *workflow-level* group instead: a later push's entire run — `ci` included — now can't start until an earlier push's `docker-promote` (the last step of its run) has fully finished. This removes the cross-job queue-ordering argument entirely rather than relying on it, at the cost of a later push's `ci` no longer being able to overlap with an earlier push's `docker-promote`.

## Consequences

- Because `imagetools create` doesn't invoke `metadata-action` again, the `next`-tagged image keeps whatever OCI labels were baked in during the original PR build — `org.opencontainers.image.version` on `next` reads `pr-95` (or whichever PR), not literally `next`, and `.revision`/`.created` reflect the PR build's commit and timestamp, not the merge commit. This is intentional: the labels honestly describe the exact artifact that was built, consistent with retagging instead of rebuilding.
- If the GitHub REST API call ever returns more than one associated PR for a commit (possible in unusual merge topologies), the job takes `.[0]` — the first result — without further disambiguation. This hasn't been a problem in this repo's straightforward single-PR-per-merge-commit history but would need revisiting if that changes.
- A burst of merges queues each push's entire `ci` + `docker-promote` pipeline behind the previous one, rather than collapsing to just the latest or letting independent stages overlap across pushes; `timeout-minutes` per run (5 for `ci`, 10 for `docker-promote`) means a long enough burst could in theory back up substantially. This hasn't been a problem at this repo's current merge cadence.
- `docker-promote` implicitly requires `ci` to *succeed*, not just complete, since `needs: ci` only unblocks on success by default — if `push-main.yml`'s `ci` re-validation genuinely fails on the merged tree, promotion is correctly skipped too. Only cancellation (a scheduling artifact, not a code problem) needed fixing; failure continuing to block promotion is intentional.
